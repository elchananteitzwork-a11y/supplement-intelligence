Design spec — accessibility, responsive, motion (read before implementation)

Contrast

Body copy on the glass panels here is rgba(251,247,238,.72–.85) over a scrim that reaches ~90% opacity black by mid-scroll — spot-checked contrast ratio is ≥4.6:1, comfortably above WCAG AA (4.5:1) for the 13px body sizes used throughout. The one place this needs a real contrast audit before ship is the hero, where copy sits directly over the photograph at full "full" intensity before the scrim reaches its darkest stop — recommend keeping the hero on intensity="calm"-equivalent grading (as built here) rather than "full," specifically because this is a reading/trust screen, not a marketing hero.

Keyboard navigation

Every interactive element here (section-nav pills, blade legend links, nav actions) is a real <a>, not a div-with-onclick — tab order follows document order (nav → section pills → hero blades → kill criteria → each report section in reading order). The blade legend must keep CandidateCoreHero.tsx's existing pattern of being real DOM buttons/links independent of whether the WebGL canvas mounts — this prototype's rotor SVG is non-interactive by design (decorative only, aria-hidden would be added to the glow/spin layers in implementation, keeping only the rotor's own accessible label on the outer svg). The new pointer-tracked "magnetic" rotor interaction (see below) is a hover/mousemove-only enhancement layered on top of that same non-interactive, decorative SVG — it changes nothing about tab order or what's keyboard-reachable.

prefers-reduced-motion

Ken Burns drift, the rotor's slow spin, the pulsing glow, the particle drift, the report-section scroll-reveal, the blade stagger-in, the Evidence-Breadth chip stagger, and the rotor's magnetic pointer-tracking all read @media (prefers-reduced-motion: reduce) and stop outright (see the CSS block above) — same policy as the real AmbientParticles.tsx, which doesn't render particles at all under reduced motion rather than rendering them frozen. Under reduced motion, every .reveal and .chip element is forced to its fully-visible end state immediately via the media query itself (not dependent on the IntersectionObserver ever firing), so nothing is ever left permanently hidden at opacity:0. No content depends on any animation to be understood — every real number is present at rest, and the confidence/blade instruments never hide their values behind hover.

Mobile / narrow viewport

Below 640px: the blade grid drops from 3 to 2 columns; the section pill nav becomes a horizontally scrollable strip (already true of the live SectionNavMobile component this prototype's pill row is modeled on — a contained, intentional scroll strip, not page-level horizontal overflow); row labels shrink and truncate; panel padding tightens; the rotor and score number step down in size alongside it so the enlarged hero still fits comfortably. Not yet designed in this pass: a true mobile teardown of the Evidence & Confidence section's multi-column witness-dot rows, which will need its own narrow-viewport pass before implementation — flagged as an open item in the report below, not resolved here.

Reading order vs. WebGL rotor

Matching the real CandidateCoreHero.tsx architecture note: every real number (score, decision, confidence, blade magnitudes, kill criteria) must be real, always-present DOM regardless of whether a WebGL canvas ever mounts on top of it. Nothing in this prototype's hero markup is canvas-only content — the SVG rotor is a decorative enhancement layered over real DOM, not a replacement for it.

2026-07-22 polish pass — implementation notes

Background image: swapped to the owner's real night photograph (public/ambient/candidate-detail-cathedral-of-palms-night.jpg); the filter was reduced from a heavy day-to-night fake grade to a barely-there enhancement since the source is already a correctly-exposed night shot — see the comment directly above .world-img for the exact reasoning and values. Nav/section-nav: rebuilt on the glass-plane recipe instead of a flat dark bar; padding/heights kept identical to the previous version so the sticky offset math between the proto-banner, nav, and section-nav is unchanged. Report rhythm: gap tightened from 56px to 22px with a small real (non-absolutely-positioned) .rs-link connector between adjacent sections — deliberately real DOM rather than an absolutely positioned pseudo-element spanning the whole column, to avoid any negative z-index / stacking-context risk against the ambient world layers underneath. Motion: two named tiers now exist as CSS custom properties — --dur-hover-sm / --ease-soft (150-200ms, ~1px lift) for buttons, pills, and blade tiles, and --dur-hover-md / --ease-cine (200-300ms, -4px lift + scale 1.02 + shadow growth) for report panels — plus exactly one complex/"magnetic" tier interaction, reserved for the rotor (see the inline script near the end of the body), consistent with the design-intelligence pass's explicit warning against scattering magnetic effects across multiple elements.
