# R&D — UIv2-M2 Phase 1: Candidate Detail Core/Hero Summary Layer (`/memo/[id]`)

Owner-approved milestone, build order item 1 of the v2 design phase (approved mockup: `hero3d-prototype` sandbox — Core rotor, accordion evidence layer, Pull mechanic, frozen 5-word vocabulary). This document covers **Phase 1 only**: a new top-of-page WebGL Core/hero summary layer, sourced entirely from real fields. Phase 2 (re-skinning the 15 existing `components/memo/*.tsx` detail sections to `pi-*`) is a separate future milestone, not started here.

**Hard constraint (owner directive, this session):** preserve the decision engine exactly as-is. No changes to verdict logic, confidence calculations, or kill-switch evaluation. This phase only wires the approved UI to real data it reads — it does not compute anything new. If any point below is found to require a decision-engine change to render honestly, implementation stops and that point is raised for a separate decision before any such change is made.

## 1. Reuse audit

- **Score + decision**: `computeGroundedScore(m)` (`lib/scoring.ts`) → `{ score: 0-100, decision: BuildDecision, insufficientEvidence, evidenceBreadth, dimensions }`. Already the sole source for the primary pill on the current page (`components/memo/CurrentSignal.tsx`, `PILL_CFG` labels: 'Entry Supported'/'Validation Required'/'Not Supported'/'Category Creation'). Reused verbatim — the Core hub's score number and primary verdict word are this value and nothing else.
- **Confidence**: `computeConfidenceAssessment(grounded)` (`lib/confidence/independence.ts:111-141`) → `ConfidenceAssessment.overallConfidence: number | null` (0-1, weakest-link `min()`, never averaged, `lib/confidence/types.ts:59-77`) + `weakestDimension`. Reused verbatim for the Core's luminosity-as-confidence treatment (per the locked design DNA: "luminosity=confidence").
- **Per-blade data**: `ScoreDimension[]` (`lib/scoring.ts:22-31`, weights `lib/scoring.ts:255-262`) — `demand`, `marketAccessibility`, `profitability`, `consumerPain`, `virality`, `subscription` (6 weighted; `manufacturing` is qualitative-only at weight 0, excluded from the rotor's 6 blades, consistent with the existing weighting). Each carries `rawScore` (0-10) or `qualitativeLevel`, `source: 'verified'|'synthesized'`, `sourceLabel`. Reused verbatim as blade emissive-magnitude + provenance-color input.
- **Kill switches**: `KillCriterion[]` (`lib/kill-criteria.ts:20-29`, `computeKillCriteria` lines 51-93) — `{ key, label, metric, comparator, threshold, valueAtGeneration }`. Reused verbatim for the flat per-criterion statement. `evaluateKillCriterion(criterion, freshValues): boolean` (lines 100-121) reused verbatim, called only in the already-existing Watchlist context (`lib/watchlist/recheck.ts:140`) — not re-implemented, not called from a new code path in this phase except to read whatever Watchlist state already exists for this analysis, if any.
- **Auth/fetch/ownership**: `app/memo/[id]/page.tsx:12-23` (Supabase `analyses` select + ownership check) — untouched, new layer consumes the same already-fetched `Analysis`/`MemoData` object.
- **Visual/interaction reference**: `hero3d-prototype` sandbox (rotor geometry/rotation model, `usePullGesture`/`pullPhysics.ts` resistance math, accordion `grid-template-rows` expand pattern, warm-cream/gold `pi-*` palette already shipped in `tailwind.config.ts` and used by `components/pi/PipelineView.tsx`/`CandidateRow.tsx`). Ported as interaction/visual language only — its illustrative data and its 6-evidence-source `direction` (Supports/Opposes/Mixed) model are explicitly NOT ported (no real field carries directional stance; see Risks).

## 4. Risks

- **No real "direction" (Supports/Opposes/Mixed) field exists.** The prototype's 6 evidence-source cards each had one; real `ScoreDimension` only carries magnitude (`rawScore`) + provenance (`source`/`sourceLabel`), never a stance. Resolution: blades are driven by magnitude + provenance only, never a fabricated stance — this is a **visual/interaction change from the approved mockup**, not a data-integrity workaround, and is called out explicitly here rather than silently resolved in code.
- **Blade-click → section navigation has no clean 1:1 mapping.** The prototype's blade-click scrolled to a matching evidence card by id. Real `MemoDisplay.CORE_SECTIONS` (`CurrentSignal`, `DemandIntensity`, `SupplyLandscape`, `UnitEconomicsTable`, `DifferentiationBrief`, `StrategicReadinessChecklist`, `KillCriteriaList`) don't correspond 1:1 to the 6 `ScoreDimension`s (e.g. `profitability` and `consumerPain` have no single named section to jump to). **Open question, not resolved here**: either (a) blade click scrolls to the nearest reasonable existing section per dimension (a judgment call to make and document explicitly at implementation time, one mapping decision per dimension), or (b) blade click is deferred to Phase 2 once sections are restructured. Flagging for implementation-time decision, not deciding here.
- **Kill-switch "Watching"/"Triggered" state does not exist for a non-watchlisted analysis.** `evaluateKillCriterion` only ever runs in the Watchlist re-check path. This phase must render the real flat `valueAtGeneration` statement for every analysis, and may show a real triggered/not-triggered pill **only** when the analysis is confirmed on the user's watchlist with real `freshValues` already computed by the existing recheck pipeline — never a client-side re-evaluation, never a fabricated state for a non-watchlisted analysis.
- **Two non-agreeing verdict systems already coexist by design** (legacy `decision` pill, primary; V2 `market_verdict`/`opportunity_quality`, secondary/collapsed — `CurrentSignal.tsx:129-140`). The Core hub shows the legacy `decision`/score only, matching the current primary/unconditional display — it does not attempt to reconcile or replace the secondary V2 verdict, which stays exactly where and how it is today (behind "Show technical detail →", untouched).
- **New heavy dependencies in production** (three.js ecosystem) — bundle-size and first-load-JS impact on a page with no prior WebGL. Mitigate via dynamic import / code-splitting the Core component (matches the existing `hero3d-prototype` build's own >500kB chunk warning, seen in that sandbox's `vite build` output this session).
- **Zero component-render test coverage exists for this page** (confirmed by architecture-agent) — regression risk is real but not test-catchable; verification is manual/visual, not automated for the rendering layer.

## 5. Testing plan

`npx tsc --noEmit` and `npm run build` must both pass. `field-derivations.test.ts`, `lib/__tests__/kill-criteria.test.ts`, `lib/watchlist/__tests__/recheck.test.ts` must stay green unmodified (their function contracts are not touched). Manual verification: open a real existing analysis's `/memo/[id]` and confirm every number the Core layer shows (score, decision word, confidence, per-blade magnitude) matches the number already rendered further down the page by the untouched legacy components for that same analysis — this is the concrete "output matches current production logic" check the owner asked for, done by direct visual comparison on a real record, not by asserting against a mock.

## 6. Smallest-correct-scope

The Core/hero layer renders: real score + real decision word (legacy pill only), real confidence (weakest-link, as-is), 6 blades driven by real `rawScore`/`source` per `ScoreDimension` (no direction), real flat kill-criteria statements (+ real triggered/not-triggered pill only if genuinely watchlisted). Pull mechanic and blade-click may ship in a reduced/honest form (e.g. Pull's resistance math re-derived from real magnitude+source instead of the prototype's illustrative direction weights — itself a follow-up data-mapping decision, not assumed solved by this document). Everything below the new layer (`MemoDisplay` and its 15 children) renders exactly as it does in production today, untouched.

## 7. Non-goals

No re-skin of the 15 existing `components/memo/*.tsx` sections (Phase 2). No change to `computeGroundedScore`, `computeConfidenceAssessment`, `evaluateKillCriterion`, or any decision-engine function's logic, thresholds, or output — if any honest rendering need surfaces that would require such a change, implementation stops and it's raised separately, per owner directive. No fabricated "direction"/stance data anywhere. No new client-side kill-criteria re-evaluation outside the existing Watchlist recheck path. No reconciliation of the two non-agreeing verdict systems — the secondary V2 verdict stays exactly as collapsed/secondary as it is today. No AppShell migration (tracked separately, backlog).
